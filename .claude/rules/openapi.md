## OpenAPI

### Architecture

- **Spec version**: OpenAPI 3.0.3 (not 3.1) -- required for oasdiff compatibility. Uses `OpenApiGeneratorV3`
- **Two-pass generation**: (1) filesystem scanner auto-infers schemas from factory `.config` on route handlers, (2) `scripts/merge-openapi-supplements.ts` overlays human-authored metadata
- **`public/openapi.json`** is gitignored -- regenerated by `prebuild:next` on every build

### Commands

```bash
npx tsx scripts/generate-openapi.ts                   # Regenerate spec
oasdiff changelog old.json new.json --format markdown # Local changelog diff
```

### CI Changelog

- Workflow: `.github/workflows/openapi-changelog.yml`
- Uses `go install github.com/oasdiff/oasdiff@latest` + direct CLI (not Docker action)
- Spec files in `.oasdiff/` (Docker actions can't see `/tmp`)
- `docs/API_CHANGELOG.md` is CI-generated, ignored by oxfmt + markdownlint

### Factory `.config`

All handler factories expose `.config` for the OpenAPI scanner:

**V1 factories** (`create-handler.ts`): `.config` has `{ factoryName, inputSchema, outputSchema }`. The scanner detects auth via `factoryName.includes('WithAuth')`.

**V2 factories** (`create-handler-v2.ts`): `createAxiomRouteHandler` passes `.config` through from the inner `withAuth`/`withPublic` layer. The `.config` shape is: `{ auth: boolean, contract: "v2", factoryName: "withAuth"|"withPublic", inputSchema, outputSchema, route }`. The scanner detects v2 factories via `config.contract === "v2"` and uses bare response schemas (no `{data, error}` envelope). For v2 route authoring patterns, see `api-v2.md`.

**Non-factory routes** can be scanner-discoverable via `Object.assign(handleGet, { config: { ... } satisfies HandlerConfig })`.

`SKIP_PATHS` in `generate-openapi.ts` -- remove entries when migrating routes to be scanner-discoverable.

### Supplement Files

- Located in `src/lib/openapi/endpoints/<domain>/` -- data-only `EndpointSupplement` exports (NOT `registry.registerPath()`)
- Provide: tags, summary, description, security, examples, `additionalResponses`, `response400Examples`
- Example data extracted to colocated `-examples.ts` files when endpoint exceeds 250 lines
- Named examples: kebab-case keys, `summary` + `description` on ALL examples, happy paths first

### New Endpoint Checklist

1. Create `route.ts` with factory handler + `schema.ts` with Zod schemas
2. Create supplement file with metadata -- scanner handles registration
3. Regenerate spec and verify at `/api-docs`

### New Supplement Domain

1. Create `src/lib/openapi/endpoints/<domain>/` with supplement + barrel
2. Add `import * as <domain>Supplements` + `allModules` entry in `scripts/merge-openapi-supplements.ts`
3. Add `export * from "./<domain>"` in `src/lib/openapi/endpoints/index.ts`

### Response Components

`ValidationError` (400), `Unauthorized` (401), `InternalError` (500) -- registered as `$ref` in `generate-openapi.ts`. The merge script preserves `$ref` for non-400 codes; 400 `$ref` inlined when `additionalResponses` targets it.

### Shared Schemas

- `src/lib/openapi/schemas/shared.ts` -- registered as `$ref` entries
- Raw `SchemaObject` (edge functions only): use `registry.registerComponent("schemas", "Name", { ... })` in `registry.ts`

### Edge Functions

- 3 `edge-process-imports.ts` files are the ONLY files using `registry.registerPath()` -- raw SchemaObject, not Zod
- Security: use `serviceRoleAuth` (not `bearerAuth`)
- Per-path `servers` override from `edge-function-servers.ts`

### Security

- `[{ [bearerAuth.name]: [] }, {}]` -- empty `{}` means cookie auth also accepted (intentional)
- Public endpoints must include `security: []` to prevent inheriting global security
